# User-Agent Pass-Through Design

## Overview

Add inbound `User-Agent` pass-through for file proxy requests so the proxy forwards the client's `User-Agent` to upstream file endpoints.

This change applies to both HuggingFace and ModelScope file flows. It does not add a proxy-owned default `User-Agent`, and it does not change the existing metadata probe strategy.

The immediate motivation is a confirmed ModelScope CDN failure mode where redirect-target requests can return `403` when the upstream request lacks an acceptable `User-Agent`.

## Goals

- Keep file requests transparent to clients.
- Continue following upstream redirects internally.
- Preserve existing metadata and caching behavior.
- Minimize blast radius by changing request header propagation only.
- Use one consistent approach for HuggingFace and ModelScope file proxy requests.

## Non-Goals

- No new default `User-Agent` generated by the proxy.
- No metadata probe redesign.
- No changes to database schema or cache identity.
- No changes to non-file API proxying.

## Root Cause

ModelScope upstream has been observed denying redirected CDN requests when the request lacks an acceptable `User-Agent`.

Current file proxy code constructs upstream requests from server-owned clients and does not propagate the inbound `User-Agent` from the client request. As a result, the proxy can send an upstream request fingerprint that differs from a successful direct client request.

For the reported failure, this is sufficient to explain the observed `403` on the ModelScope file route.

## Design

### 1. Capture inbound User-Agent on file proxy routes

For file proxy handlers, read the incoming `User-Agent` header from the client request.

Requirements:

- Apply to both HuggingFace and ModelScope file routes.
- Only propagate when the incoming request actually contains `User-Agent`.
- Do not synthesize a fallback value.

File routes in scope:

- `/{org}/{repo}/resolve/{revision}/{*path}`
- `/api/resolve-cache/{repo_type}/{org}/{repo}/{revision}/{*path}`
- `/hf/{org}/{repo}/resolve/{revision}/{*path}`
- `/ms/{org}/{repo}/resolve/{revision}/{*path}`
- `/ms/api/v1/models/{org}/{repo}/repo`

### 2. Forward User-Agent on all upstream file requests triggered by that client request

Once captured, the inbound `User-Agent` should be attached to all upstream file requests made on behalf of that incoming request.

This includes:

- direct `GET` requests to upstream file URLs
- direct `HEAD` requests to upstream file URLs
- redirected follow-up requests to final file URLs when the proxy performs the follow itself
- range requests issued while downloading chunks from upstream

The main rule is: if a given outgoing upstream request exists because of one inbound client file request, it should carry the same `User-Agent` if one was provided.

### 3. Keep provider-neutral behavior

The header propagation mechanism should be shared by HF and MS file paths instead of being provider-specific.

Why:

- same implementation shape for both providers
- smaller surface area than a ModelScope-only special case
- easier to reason about request parity between direct client access and proxied access

### 4. Keep existing metadata logic unchanged

This change does not redesign how metadata is discovered.

Current behavior remains:

- upstream `302` is followed internally
- client receives final `200` or `206`
- existing `HEAD` and cache metadata flow stays as-is
- existing service/session download flow stays as-is

The only intended behavioral change is that upstream file requests can now include the caller's `User-Agent`.

### 5. Preserve strict header rules

The proxy should only pass through `User-Agent` when it was present on the inbound request.

Rules:

- do not invent a fallback `User-Agent`
- do not overwrite an explicitly configured non-file request policy
- do not start forwarding unrelated headers as part of this change

## Affected Code

### `src/server.rs`

- Capture inbound `User-Agent` in file route handlers.
- Pass the captured header through to downstream helper paths.

### `src/service.rs`

- Thread optional `User-Agent` through upstream file request code paths.

### `src/session.rs`

- Thread optional `User-Agent` through session-based metadata and chunk download requests.

## Implementation Shape

Prefer a small, explicit parameter for optional inbound `User-Agent` rather than global mutable request context.

Recommended shape:

- extract `Option<String>` from the inbound request headers in `src/server.rs`
- pass that optional value into file-serving paths in `src/server.rs`, `src/service.rs`, and `src/session.rs`
- when building a `reqwest::RequestBuilder` for upstream file access, conditionally add `.header("User-Agent", ua)`

This keeps the change explicit and request-scoped.

## Error Handling

- If the inbound request has no `User-Agent`, proceed exactly as today.
- If the upstream still returns `403`, existing error handling stays in place.
- Logging may include whether a `User-Agent` was present, but should not log the full header value unless already consistent with project logging standards.

## Testing

Add focused tests covering request header propagation for file requests.

### Required cases

1. A file proxy request with inbound `User-Agent` forwards that header upstream.
2. The forwarded `User-Agent` is present on redirected follow-up requests.
3. Session-based chunk downloads also forward the same `User-Agent`.
4. A request without inbound `User-Agent` preserves current behavior and does not inject one.

### Test style

- Use local mock upstreams rather than live HF or MS.
- Verify received upstream request headers directly in the mock server.
- Include at least one redirecting case and one chunk/range download case.

## Risks

- The `User-Agent` needs to be threaded through multiple file-fetch layers, including session-based downloads.
- It is easy to fix only top-level `HEAD` or `GET` and miss chunk downloads in `session.rs`.
- This change does not solve upstream behaviors unrelated to `User-Agent`, such as provider-specific `HEAD` quirks.

## Alternatives Considered

### 1. Proxy-owned default User-Agent

Pros:

- works even when the inbound client did not send `User-Agent`

Cons:

- changes proxy identity rather than preserving client request identity
- adds a policy decision the user explicitly does not want for this change

Rejected for this change.

### 2. ModelScope-only User-Agent handling

Pros:

- narrower scope

Cons:

- duplicates logic that can be shared by HF and MS
- gives different request semantics to two otherwise similar proxy paths

Rejected in favor of one provider-neutral file proxy rule.

## Success Criteria

- HF and MS file proxy requests forward inbound `User-Agent` when present.
- Redirected upstream file requests also receive the same `User-Agent`.
- Existing file proxy behavior is unchanged when the client did not send `User-Agent`.
- New tests reproduce the missing-header behavior and pass with the fix.
